# 語魂組織治理規格（Human Governance, v1.1）

日期：2026-02-11  
狀態：`authoritative`（人類治理規格）  
依賴：`01_術語與門檻單一規格.md`

## 0. 範圍聲明

本文件只描述人類決策與版本治理流程，不描述 runtime 程式邏輯。

## 1. 治理角色

1. Maintainer
- 維護主幹規格與版本發布
- 裁定術語衝突與遷移策略

2. Reviewer
- 檢查設計一致性、風險與回滾可行性

3. Auditor
- 驗證「文件宣稱」與「repo 實作」是否一致

## 2. 提案到決策流程

1. Proposal（提案）
- 必填：動機、影響面、風險、回滾方案、測試證據

2. Review（審查）
- 至少一位 Reviewer + 一位 Auditor
- 先做一致性審查，再做風險審查

3. Decision（決策）
- 一般變更：簡單多數
- 高風險變更（P0/Gate/審計格式）：`2/3` 多數

4. Trace（追溯）
- 決策需有編號、日期、責任人、版本標記

## 3. 變更分級（SemVer 對應）

- Patch：文字澄清、排版、無行為變更
- Minor：新增可相容欄位/流程，既有行為不破壞
- Major：改動核心語義（如 `ΔS/ΔΣ`、POAV、P-level）

規則：

1. Major 必附 `MIGRATION_NOTE`。  
2. 所有版本必附 `CHANGELOG`。  
3. 若牽涉 runtime 規則，需附最小可重現測試證據。

## 4. 否決與緊急機制

1. Delayed Veto（延遲否決）
- 決策後 `72` 小時內可提出
- 必須附可驗證風險證據

2. Emergency Freeze（緊急凍結）
- 觸發：發現 P0 漏洞、重大誤導輸出、不可追溯回歸
- 動作：停止發布、回退上一穩定版本、啟動事故審查

## 5. 文件治理硬規則

1. Runtime/Human 不可混層：  
- runtime 規則寫在 `02_執行治理規格_Runtime.md`  
- 組織流程寫在本文件

2. 術語單一來源：  
- `01_術語與門檻單一規格.md` 是唯一術語源

3. 路徑真實性：  
- 文檔中的程式路徑必須 `Test-Path == True`

4. 禁止雙定義：  
- 禁止同時存在兩套 POAV / P-level / 向量語義

## 6. 發版前審核清單（Human）

1. 術語一致性
- 是否全部遵循 `01`？

2. 路徑一致性
- 文檔引用路徑是否可在 repo 找到？

3. 測試證據一致性
- 宣稱的行為是否有對應測試或 smoke 腳本？

4. 回滾可行性
- 是否能快速回退到上一版並復原服務？

## 7. 與 CI 的最小對齊要求

提交規格變更時，至少應對齊：

1. Python 測試與 lint  
2. Web lint/test/build  
3. Web+Backend smoke（`scripts/verify_web_api.py`）  

說明：CI pipeline 會演進，但治理文件不可脫離最小可重現檢查。

## 8. 事故後檢討（Postmortem）模板

每次重大事故至少記錄：

1. 事件時間線（含首次發現時間）  
2. 觸發規則與漏檢點  
3. 使用者影響面  
4. 即時止血動作  
5. 長期修復方案（含 owner 與 deadline）
