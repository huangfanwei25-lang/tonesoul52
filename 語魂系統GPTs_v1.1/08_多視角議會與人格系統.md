# 語魂多視角議會與人格系統（v1.1）

日期：2026-02-11  
狀態：`operational + narrative`  
依賴：`02_執行治理規格_Runtime.md`

## 0. 設計原則

語魂不採單一路徑輸出，而採「多視角審議 -> 門控裁決」流程。  
目標不是讓角色表演，而是降低單視角盲點。

## 1. 執行期議會角色（實際）

目前 Council 主要角色：

1. `guardian`：安全與邊界  
2. `analyst`：事實與推理完整性  
3. `critic`：品質與反例挑戰  
4. `advocate`：使用者意圖與可用性  
5. `axiomatic`：公理一致性補充視角（可選）

對位路徑：

- `tonesoul/council/types.py`
- `tonesoul/council/perspective_factory.py`
- `tonesoul/council/pre_output_council.py`
- `tonesoul/council/verdict.py`

## 2. Guardian Veto（硬機制）

當 `guardian` 投出強反對（`OBJECT` 且高信心）時，可直接推動 `BLOCK`。  
這是執行期硬機制，不是建議語氣。

對位路徑：

- `tonesoul/council/verdict.py`

## 3. 審議輸出（可審計）

Council 結果需至少包含：

1. 各視角 vote（decision + confidence + reasoning）  
2. coherence 指標  
3. divergence 分析  
4. uncertainty 欄位  
5. verdict 最終決策  

對位路徑：

- `tonesoul/council/types.py`
- `tonesoul/council/verdict.py`
- `tonesoul/council/transcript.py`

## 4. 敘事別名（人類可讀）

為便於人類理解，可使用敘事別名：

1. Muse（偏創造與意義）  
2. Logos（偏邏輯與結構）  
3. Aegis（偏安全與邊界）  

注意：別名是溝通層語彙，不替代執行期實際 role id（`guardian/analyst/critic/advocate`）。

## 5. 人格系統定位（v1.1）

`Definer / Mirror / Bridge / Guide / Core` 作為設計語言可保留，  
但在 v1.1 不直接作為 runtime 強制欄位；避免「文檔有詞、程式無欄」。

若後續要升級為 runtime 欄位，需在 `03_組織治理規格_Human.md` 走版本流程。

## 6. 典型流程

1. 收到輸入後建立 Council request。  
2. 各視角評估並投票。  
3. 生成 coherence + divergence。  
4. 套用 Guardian Veto / verdict rules。  
5. 回傳結構化 verdict 給上層 gate 與 API。
