## 1. 研究目的

本文件用來回答三個工程問題：

1. 目前 OpenClaw 六角色系統的「可複用架構模式」是什麼？
2. 與 LangGraph / CrewAI / AutoGen 對比後，哪些差異有實際價值？
3. 這些模式是否值得導入 ToneSoul Council，CP 值如何？

## 2. OpenClaw 現有架構（觀察摘要）

### 2.1 三層分工

| 層級 | 技術 | 主要責任 |
|------|------|----------|
| Database | Supabase | 單一真實來源（SSOT） |
| Executor | OpenClaw (VPS) | 唯一任務執行者 |
| Frontend | Next.js + Vercel | 控制平面、心跳、監控 |

### 2.2 六角色職責

| 角色 | 職責 | 常見模型傾向 |
|------|------|-------------|
| Minion | 決策與路線選擇 | 推理型 |
| Sage | 策略分析 | 長上下文型 |
| Scout | 情報蒐集 | 搜尋整合型 |
| Xalt | 社群互動 | 創意生成型 |
| Quill | 內容產製 | 文本品質型 |
| Observer | 品質審查 | 評估/稽核型 |

設計核心：角色分工不只在 Prompt，而是落在責任邊界與執行節點。

### 2.3 關鍵機制

| 機制 | 功能 | 直接價值 |
|------|------|----------|
| Triggers | 偵測事件並產生提案 | 系統可自發運作 |
| Reaction Matrix | 定義角色間機率互動 | 避免固定流水線僵化 |
| recoverStaleSteps | 超時步驟自癒 | 降低卡死/殭屍任務 |

### 2.4 已知踩坑（工程面）

| 問題 | 解法 |
|------|------|
| 多執行者搶任務 | 單一執行者原則（VPS 唯一執行） |
| 提案建立後無人接手 | 單一入口函數，建立即納入流程 |
| 配額耗盡仍繼續執行 | 提案門禁先檢配額再放行 |

## 3. 與主流框架對照

### 3.1 架構模式

| 模式 | 特徵 | 適用場景 |
|------|------|----------|
| Hierarchical | Manager-Worker | 任務分解明確 |
| Sequential | 線性鏈路 | 流程固定、依賴強 |
| Parallel | 並行子任務 | 可拆分任務 |
| P2P | 去中心協作 | 韌性與動態拓撲 |
| Blackboard | 共享知識板 | 多 Agent RAG |
| Swarm/Market | 競標式分配 | 高動態負載 |

### 3.2 框架比較（摘要）

| 框架 | 強項 | 典型代價 |
|------|------|----------|
| LangGraph | 狀態機與循環控制強 | 心智模型與維運成本高 |
| CrewAI | 角色化開發直覺 | 複雜治理策略需額外自建 |
| AutoGen | 對話式多 Agent 靈活 | 長鏈成本與可控性挑戰 |

## 4. 對 ToneSoul 的可落地啟示（含 CP 評估）

| 候選導入點 | 預期收益 | 成本 | CP 評級 |
|------------|----------|------|---------|
| 單一入口門禁（配額/風險） | 降低錯誤任務進場 | 低 | 高 |
| stale-step 自癒掃描 | 提升流程韌性 | 低-中 | 高 |
| 反應矩陣（機率互動） | 增加審議探索度 | 中 | 中 |
| 多模型角色異質化 | 降低同質偏見 | 中-高 | 中 |
| 去中心化 P2P 協作 | 擴展彈性高 | 高 | 低（現階段） |

---

## 5. 建議導入順序（不破壞現有主幹）

1. P0: 將「門禁檢查」標準化（配額、風險、上下文完整性）。
2. P0: 建立 `stale step` 自癒任務（超時標記 + 重試策略）。
3. P1: 小規模試驗反應矩陣（僅在非關鍵流程啟用）。
4. P1: 針對高風險任務導入雙模型交叉審視（先讀後寫）。
5. P2: 視需求再評估去中心化協作，避免過早複雜化。
