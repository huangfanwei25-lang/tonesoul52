# V1.0.0 Stable Release Post-Mortem & Context

**Date**: 2026-02-25
**Version**: `v1.0.0-stable` (Commit: `08a19ca`)
**Status**: First Normal Release (MVP)

## 📌 Executive Summary
This document serves as the permanent memory for how the ToneSoul UI Chat successfully reached its first stable, working state on Vercel. Prior to this release, the production environment suffered from complete blocking (500/404 errors) during the Synthesizer phase of the chat deliberation, and severe "clogging" due to API rate limits.

## 🐛 The Three Critical Blockers & How We Fixed Them

### 1. The "404 Not Found" AI Model Expiration
- **Context**: The Synthesizer phase was hardcoded to use `gemini-2.5-flash-preview-05-20` in `ChatInterface.tsx` (`callGeminiDeepThinkingAPI`). Google deprecated and removed this preview model.
- **Symptom**: Every chat request failed silently in the frontend, resulting in a dead UI.
- **Resolution**: Updated all Gemini API calls to use the General Availability (GA) model: `gemini-2.5-flash`.

### 2. The Vercel Serverless Cache Sabotage
- **Context**: The Vercel deployment relies on the Python backend in `api/chat.py`, which imports shared utilities from `api/_shared/core.py`.
- **Symptom**: `core.py` contained 28 lines of duplicate "dummy stub" functions at the very bottom of the file (e.g., `def _chat_cache_get... return None`). Because Python reads top-to-bottom, these stubs silently overwrote the actual working caching logic defined earlier in the file.
- **Resolution**: Cleaned up the file by deleting the duplicate stubs (lines 533-559), restoring the proper logic pipeline.

### 3. The "429 Too Many Requests" Concurrency Clog
- **Context**: The `performMultiPathDeliberation` function in React used `Promise.all` to simultaneously fire three massive LLM prompts (Philosopher, Engineer, Guardian) in the exact same millisecond. 
- **Symptom**: Free-tier or low-tier API keys (Gemini/OpenAI) strictly limit requests to 1-2 per second. The simultaneous burst caused immediate `429 Too Many Requests` bans, blocking the chat and creating a feeling of "congestion".
- **Resolution (Phase 1 Fix)**: Refactored the `Promise.all` into sequential `await` calls. The UI now loads progressively (`哲學家思考中...` ➔ `工程師評估中...` ➔ `守護者審查中...`).
- **Trade-off**: Time-to-first-token is slightly slower, but 429 rate limit errors drop to zero, ensuring 100% successful execution on free APIs.

## 🛡️ Red Team Audit Discoveries (QA Auditor)
Prior to release, we delegated a chaos-engineering audit to the local Qwen3:4b model (via the `qa_auditor` skill) against `api/_shared/core.py`. 

**Known Vulnerability Accepted for V1**:
- **D3 JSONL Race Condition**: The backend `JsonlSoulDB` writes to a shared JSONL file without explicit thread/process locks. Under high parallel load (50+ concurrent users), interleaved text writes will corrupt the JSON payload. 
- **D4 Ephemeral Wipe**: Vercel Serverless spin-downs will wipe any side-effects not explicitly sent to a persistent DB (like Supabase).
- *Decision*: Acceptable for V1 MVP, as current traffic does not hit concurrent upper limits. 

## 🚀 Next Steps (Architecture Roadmap)
To resolve the sequential speed trade-off introduced in Blocker #3, the next architectural leap should be the **Wei Xiaobao Protocol (Phase 2)**:
1. Move the 3-way LLM calls out of the React frontend (`ChatInterface.tsx`) entirely.
2. Push them to the Python backend (`api/chat.py`).
3. Use Python `asyncio` to manage rate limits elegantly.
4. Stream the responses back to the frontend via Server-Sent Events (SSE) so the user sees a "typewriter effect" instantly as the personas debate.
