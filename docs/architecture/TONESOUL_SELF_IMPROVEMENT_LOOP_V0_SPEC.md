# ToneSoul Self-Improvement Loop v0 Spec

> Purpose: define a bounded, testable self-improvement loop for ToneSoul that can optimize selected operator/runtime surfaces without collapsing into mythology, silent self-rewrite, or identity drift.
> Status: accepted architecture spec for a future bounded program. Not yet the active short board.
> Authority: architecture contract. This file constrains what a ToneSoul-native improvement loop may and may not do, but it does not itself authorize runtime mutation.

---

## Why This Exists

ToneSoul already has enough structure that the next bottleneck is no longer only:

- shared transport
- handoff discipline
- workspace entry layering

The next bottleneck is:

`can ToneSoul improve its own bounded operator/runtime surfaces without lying about what changed, why it changed, or what must stay untouched?`

This line exists now because several prerequisites are finally present:

1. `tiered entry and workspace shells` now exist
2. `consumer contracts` now exist across Codex-style, Claude-style, dashboard, and observer consumers
3. `closeout grammar` now exists, so partial/blocked work is less likely to masquerade as completion
4. `knowledge-layer boundaries` now exist, so future lessons can be compiled without reusing hot coordination state
5. `surface versioning and consumer lineage` now exist, so later readers can tell which readout shape they are looking at

Without those prerequisites, a "self-improvement loop" would mostly be fantasy.

With them, ToneSoul can attempt a bounded v0.

---

## Compressed Thesis

ToneSoul should not jump to a general "self-evolving AI" story.

ToneSoul should first build:

`one bounded loop that proposes, tests, analyzes, and promotes improvements only inside a tightly defined mutation space.`

That is the v0.

---

## What This Is

`Self-Improvement Loop v0` is a bounded improvement mechanism for:

- operator workspace surfaces
- routing and escalation hints
- consumer-parity readouts
- bounded retrieval/operator query behavior
- launch-health or validation readout packaging

It is a system for:

- observing current friction
- proposing small candidate changes
- running explicit evaluation
- distilling lessons
- promoting only when the result is measurably better and still honest

---

## What This Is Not

This is **not**:

- autonomous identity rewrite
- automatic `AXIOMS.json` revision
- automatic vow or permission mutation
- hidden-thought mining
- vendor-native "shared mind" interoperability
- a replacement for human architectural judgment
- a justification for turning R-memory into an experiment database

If it cannot stay bounded, it does not belong in v0.

## Required Inputs

The loop may read from these classes of sources:

### 1. Canonical anchors

- canonical architecture contracts
- `DESIGN.md`
- bounded control-plane and governance contracts

### 2. Runtime evidence

- tests
- validation waves
- closeout grammar outputs
- drift/readout diagnostics
- workspace-role parity checks

### 3. Compiled knowledge

- future compiled-knowledge artifacts
- research distillations that have already been rewritten into ToneSoul-native terms

### 4. Hot coordination state

- session-start
- observer window

But only as current-state orientation, not as sole proof that a mutation is good.

## Core Boundary

R-memory is:

- hot coordination
- current orientation
- bounded handoff

R-memory is **not**:

- experiment database
- durable lesson archive
- calibrated evaluation registry
- identity truth

Any future experiment registry must remain separate from hot coordination transport.

## Loop Stages

## Stage 1: Observe

Open a candidate only when there is a real signal such as:

- repeated consumer drift
- persistent operator friction
- stale or misleading readout shape
- over-escalation cost
- under-surfaced blocked/partial status

Do not open a candidate because a theory sounds elegant.

## Stage 2: Bound

Before changing anything, define:

- exact mutation surface
- out-of-scope surfaces
- evaluator bundle
- rollback path
- overclaim to avoid

If the mutation boundary is fuzzy, stop here.

## Stage 3: Design

Produce one bounded candidate with:

- small surface area
- explicit result shape
- regression plan
- promotion rule

Avoid multi-subsystem rewrites in v0.

## Stage 4: Experiment

Run the candidate against:

- targeted regression tests
- parity checks
- role/readout validation
- if applicable, a bounded workspace or operator smoke

The loop may compare multiple candidates, but only inside the same mutation class.

## Stage 5: Analyze And Distill

Every experiment must end with:

- result classification
- failure mode
- unresolved items
- whether the observed gain is descriptive, operational, or only cosmetic

This stage is where the loop earns the right to promote.

## Stage 6: Promote, Park, Or Retire

Promotion requires:

- regression green
- bounded evidence of improvement
- no new silent-promotion hazard
- no crossed authority boundary

If those are not met, the result must be parked or retired, not narrated as success.

---

## Allowed Mutation Space In v0

The v0 loop may optimize:

1. `workspace/operator shell`
   - panel ordering
   - cue phrasing
   - tier alignment
   - deep-drawer defaults

2. `routing and escalation hints`
   - deliberation-mode hinting
   - mutation preflight phrasing
   - operator walkthrough packaging

3. `consumer-parity surfaces`
   - consumer contracts
   - lineage/fallback wording
   - cross-consumer drift readouts

4. `bounded operator retrieval`
   - provenance-first result shape
   - query scope
   - non-promotion reminders

5. `launch/readout packaging`
   - descriptive-vs-trendable separation
   - overclaim guard surfaces

These are strong enough to matter and bounded enough to test.

---

## Forbidden Mutation Space In v0

The v0 loop must not mutate:

1. `canonical governance truth`
   - `AXIOMS.json`
   - core constitutional claims

2. `vows, permissions, or human authority boundaries`

3. `durable identity semantics`
   - subject identity
   - stable vows
   - durable boundaries

4. `R-memory transport semantics`
   - packet truth order
   - hot coordination as a concept
   - hidden transport assumptions

5. `vendor-native interop claims`
   - no pretending official Claude/Codex shared-memory exists

6. `opaque model-self claims`
   - no selfhood inflation
   - no emotion inflation
   - no hidden-thought dependence

---

## Human-Gated Mutation Classes

Some classes may become valid later, but remain human-gated for now:

- experiment-registry schema introduction
- predictive readout math
- backend coordination promotion
- deeper council-confidence calibration
- cross-repo or private-repo migration decisions

v0 may prepare these.
v0 may not silently execute them.

---

## Evaluator Harness Minimum Shape

Every self-improvement candidate must declare:

- `candidate_id`
- `target_surface`
- `baseline_story`
- `success_metric`
- `failure_mode_watch`
- `evidence_bundle`
- `rollback_path`
- `overclaim_to_avoid`

If a candidate cannot name these, it is not ready for experiment.

---

## Experiment Registry Boundary

ToneSoul will eventually need an experiment registry.

For v0, the rule is:

- runtime evidence can live in tests and validation artifacts
- experimental notes may live in bounded plan/status surfaces
- future registry storage belongs under the compiled-knowledge side, not hot memory

This keeps:

- hot coordination
- compiled lessons
- experiment lineage

from collapsing into one bucket.

---

## Cross-Agent Parity Rule

A candidate that improves only one shell while degrading shared interpretation does not count as a successful v0 improvement.

At minimum, promotion should consider impact on:

- Codex-style entry
- Claude-compatible adapter
- dashboard/operator shell
- observer/readout consumers when relevant

The loop is allowed to prefer one consumer.
It is not allowed to silently fracture the shared reading order.

---

## Anti-Fake-Completion Rule

The loop must never promote a candidate only because:

- the prose sounds cleaner
- the demo looks smoother
- the narrative is more confident

Promotion must still respect:

- closeout grammar
- unresolved items
- blocked or underdetermined states
- explicit failure modes

If a candidate ends `partial` or `blocked`, it must stay labeled that way.

---

## Initial Bounded Trial Set

The safest first trial families are:

1. operator workspace tier alignment
2. deliberation escalation hint quality
3. cross-consumer parity/readout consistency
4. bounded operator-retrieval packaging

These are the best v0 targets because they are:

- high leverage
- observable
- low identity risk
- easy to regression-test

---

## Exit Criteria For v0

ToneSoul can claim the v0 loop is working only when:

1. at least one bounded candidate was promoted with explicit evidence
2. at least one candidate was parked or retired honestly instead of force-promoted
3. cross-consumer parity remained intact
4. no identity/governance/vow boundary was crossed
5. the loop produced reusable lessons, not only one-off patches

Until then, this remains a bounded program, not a grand capability claim.

---

## First Required Contract

The first contract this loop depends on is:

- `TONESOUL_SELF_IMPROVEMENT_EVALUATOR_HARNESS_CONTRACT.md`

That order is deliberate.

ToneSoul should not open experiment lineage, mutation-space expansion, or trial waves before the evaluator harness exists.

The second required contract is:

- `TONESOUL_EXPERIMENT_REGISTRY_AND_LINEAGE_BOUNDARY_CONTRACT.md`

That order is also deliberate.

ToneSoul should not build a trial-memory habit until it can keep experiment lineage separate from hot coordination, compiled knowledge, and identity.

The third required contract is:

- `TONESOUL_BOUNDED_MUTATION_SPACE_CONTRACT.md`

This is the line that keeps the loop from wandering into governance, identity, vow, or transport mutation while still calling itself "bounded."

The fourth required contract is:

- `TONESOUL_ANALYZER_AND_PROMOTION_GATE_CONTRACT.md`

Without it, the loop could still evaluate and record trials, then over-smooth the closeout into false success.

The fifth required contract after the first trial wave is:

- `TONESOUL_PROMOTION_READY_RESULT_SURFACE_CONTRACT.md`

Without it, bounded trial outcomes would either stay too hidden to help later agents or leak upward into first-hop shells as if experiment history were runtime truth.
