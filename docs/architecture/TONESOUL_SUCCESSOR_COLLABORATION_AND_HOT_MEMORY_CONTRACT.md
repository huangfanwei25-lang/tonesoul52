# ToneSoul Successor Collaboration And Hot-Memory Contract

> Purpose: define how ToneSoul should externalize enough hot memory, design center, and collaboration posture that a successor AI can continue work safely without hidden chat history.
> Last Updated: 2026-04-02
> Authority: architecture and collaboration boundary aid. Does not outrank runtime code, canonical governance truth, tests, or human-managed private residue.

---

## Why This Exists

ToneSoul already has many bounded continuity surfaces:

- session-start bundle
- packet / delta / claims
- checkpoints / compactions
- subject snapshot
- working-style continuity
- observer window
- council dossier

What still goes wrong is not pure amnesia.
It is successor confusion:

- too many adjacent surfaces with slightly different stories
- summary children being read as if they outranked their parents
- advisory residue being mistaken for canonical truth
- later agents reconstructing the current center from chat history instead of repo-native surfaces

This contract exists to stop that drift.

## Compressed Thesis

Successor continuity should preserve:

- current center
- current boundaries
- current shortest board
- current honesty posture

It should not preserve:

- mythologized shared selfhood
- hidden thought
- smooth overclaim
- accidental promotion of summaries into law

## The 6 Rules

### 1. Successors Must Reconstruct From Bounded Surfaces

The default successor path is:

1. `AI_ONBOARDING.md`
2. `docs/AI_QUICKSTART.md`
3. `DESIGN.md`
4. `python scripts/start_agent_session.py --agent <your-id> --no-ack`
5. `python scripts/run_observer_window.py --agent <your-id>`
6. deeper packet / diagnose only if needed

Successors should not start from repo-wide guessing or long chat replay.

### 2. Shell / Resolver / Execution Must Stay Separate

ToneSoul should keep three orders distinct:

- `Shell`
  - entry surfaces, observer window, bounded readouts
- `Resolver`
  - readiness, receiver posture, evidence posture, low-drift anchor selection, task track, current short board
- `Execution`
  - code changes, claims, checkpoints, compactions, writes, releases

Observer surfaces may help a successor orient.
They do not directly authorize execution.

### 3. Parent And Child Surfaces Are Not Equivalent

Children must remain subordinate to their parents:

- compaction != canonical planning truth
- observer window != governance truth
- dossier summary != calibrated evaluation
- working-style playbook != identity
- packet readout != full underlying evidence

If a child surface conflicts with a higher-authority parent, the parent wins.

### 4. Advisory Surfaces Must Not Self-Promote

Successors may:

- `ack`
- `apply`
- `review`

They may not silently `promote` advisory surfaces into:

- vows or permissions
- canonical governance truth
- durable identity
- task scope or success criteria

### 5. Anti-Fake-Completion Is Mandatory

Successor-facing surfaces must not pretend completion when the real state is:

- blocked
- underdetermined
- descriptive-only
- stale
- contested

Every bounded handoff should preserve at least:

- what is done
- what is still open
- what is only advisory
- what is blocked on missing evidence or human choice

The preferred closeout grammar for resumability handoffs is:

- `complete`
- `partial`
- `blocked`
- `underdetermined`

And, when relevant:

- `stop_reason`
- `unresolved_items`
- `human_input_required`

A smooth summary without those signals must not be treated as proof that a session actually ended cleanly.

Observer-facing shells should therefore lift any latest compaction closeout that is:

- `partial`
- `blocked`
- `underdetermined`

so a successor reading the observer window first still sees that the last handoff is not equivalent to completed work.

When `closeout_attention` is lifted, it should stay bounded and action-shaped:

- `source_family`
- `attention_pressures`
- `operator_action`

This improves successor handling only.
It must not be promoted into planner behavior, stronger permissions, or governance truth.

### 6. Descriptive And Calibrated Layers Must Stay Separate

ToneSoul may expose descriptive numeric or semantic summaries.
It must not silently restate them as calibrated truth.

Examples:

- council agreement != decision accuracy
- continuity observability != shared mind
- working-style reinforcement != identity convergence

## The Hot-Memory Ladder

Successor collaboration works best when hot memory is layered, not flattened.

### Layer 0: Canonical Anchors

These are the durable center:

- `AXIOMS.json`
- `DESIGN.md`
- canonical architecture contracts
- current accepted `task.md` short board

### Layer 1: Low-Drift Anchor

This is the observer-facing center of gravity:

- current stable posture
- stable task constraints
- low-drift L1 facts
- agreed L2 definitions
- current launch truth

In current runtime/readout form, later agents should see this alongside:

- a bounded `canonical_center` derived from the accepted `task.md` short board
- a bounded `consumer_contract` that keeps Codex-style shells, Claude-style shells, and dashboard/operator shells aligned on the same first-hop read order
- a bounded source-order rule that keeps canonical anchors, live coordination truth, derived observer shells, and lower-authority handoff surfaces separate
- a bounded `hot_memory_ladder` that states which layer is parent truth and which layers are advisory children
- a bounded `hot_memory_ladder.current_pull_boundary` that says where a successor should stop pulling before deeper context becomes latency tax or authority drift
- a bounded `hot_memory_decay_map` that states which layers are operational, review-only, or quarantined
- a runtime-aligned decay/compression explanation so later agents do not invent a second ladder from sidecar notes
- a bounded `subsystem_parity` readout that states which major subsystem families are baseline, beta-usable, partial, or deferred

`current_pull_boundary` may preserve:

- `pull_posture`
- `preferred_stop_at`
- `why_now`
- `operator_action`

This remains a bounded latency and interpretation aid only.
It must not become a planner, retrieval myth, or deeper-pull permission system.

Target states:

- `stable`
- `contested`
- `stale`

### Layer 2: Live Coordination

- readiness
- task track hint
- deliberation mode hint
- repo-state awareness
- claims
- mutation preflight
- shared edit preflight
- packet freshness
- import posture
- receiver parity

### Layer 3: Bounded Handoff

- checkpoints
- compactions
- delta feed
- council dossier summary
- primary receiver alerts

### Layer 4: Working Identity

- subject snapshot
- subject refresh
- working-style anchor
- working-style playbook

This layer is inheritable but non-canonical.

### Layer 5: Replay And Review

- traces
- full council dossier
- validation waves
- status artifacts

### Layer 6: Local Residue

These remain outside shared public continuity unless explicitly reviewed:

- local sketches
- private memory data
- operator residue
- permission-locked local folders

## The Successor Reconstruction Questions

Within the first 10 minutes, a successor AI should be able to answer:

1. What is currently stable?
2. What is currently contested?
3. What is stale enough to distrust?
4. What is the current short board?
5. What may I safely apply?
6. What must I not promote?
7. What current launch tier may I honestly claim?
8. Did the repo move even if coordination delta stayed quiet?

If the current stack cannot answer those quickly, hot-memory design is not good enough yet.

## Successor Mutation Preflight

Successors should not infer mutation permission from observer prose alone.

The session-start bundle should expose one bounded `mutation_preflight` that answers:

- where shared mutation is blocked, claim-first, or locally safe
- which write lanes are already hook-guarded in runtime
- which surfaces remain human-gated
- where the next bounded write-hook follow-up belongs

This surface is advisory shell/resolver guidance.
It does not replace runtime locks, Aegis checks, or human-owned task-board ratification.

## Subsystem Parity Readout

Successors also need one bounded maturity map so they do not treat every ToneSoul lane as equally finished.

The successor-facing `subsystem_parity` readout should:

- expose only the highest-signal subsystem families
- use bounded labels: `baseline`, `beta_usable`, `partial`, `deferred`
- tie each family to one strongest current truth, one main gap, one next bounded move, and one overclaim to avoid
- keep `next_focus` subordinate to `mutation_preflight` and `canonical_center`
- when a next focus is surfaced, expose:
  - `resolved_to`
  - `source_family`
  - `focus_pressures`
  - `operator_action`

Clearer `next_focus` packaging improves successor orientation only.
It must not be read as autonomous planning, stronger mutation authority, or governance truth.

This readout is not another roadmap.
It is a successor compression surface for maturity and honesty posture.

## Non-Goals

This contract does not authorize:

- hidden-thought transfer
- one mega-prompt architecture
- total transcript preservation
- making observer window sovereign
- making working style equal identity
- making descriptive evidence equal calibrated truth

## Canonical Line

ToneSoul should make successor collaboration boringly reliable:

- easy to enter
- easy to orient
- hard to overclaim
- hard to silently promote
- easy to continue without chat-history mythology

That is what "good hot memory" means here.
