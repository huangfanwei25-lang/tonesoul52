# ToneSoul Working-Style Continuity Contract

> Purpose: define how later agents may inherit operating style, prompt discipline, and render/encoding habits without pretending that shared memory makes multiple models think as one.
> Last Updated: 2026-03-29
> Authority: continuity discipline aid. Does not outrank runtime code, canonical governance truth, or human-managed boundaries.

---

## Why This Exists

ToneSoul already shares a lot of state well:

- posture
- claims
- checkpoints
- compactions
- subject snapshots
- council dossiers

What it still does not automatically share is *working style*.

Two agents can read the same packet and still diverge on:

- scan breadth
- evidence strictness
- prompt structure
- compression habits
- shell/render interpretation
- coding cadence

The result is familiar:

`same architecture != same mind`

This contract names the bounded middle ground.
ToneSoul should not pretend to share hidden cognition.
But it can externalize a stable working style well enough that later agents stop reinventing habits from scratch.

## Compressed Thesis

Working style is inheritable, but non-canonical.
It belongs in bounded continuity surfaces, not in vows, not in posture, and not in mythologized personality.

## What Counts As Working-Style Continuity

Working-style continuity means habits such as:

- prefer `packet` before broad repo scan
- prefer explicit `P0 / P1 / P2` over vague prompt instructions
- prefer `[資料不足]` over unsupported inference
- prefer `checkpoint or compaction before release`
- treat shell `??` as a render-layer suspicion before calling a file corrupted

These are not constitutional truths.
They are durable operator habits that help later agents behave more like the same working line.

## What Does Not Count

These must not be smuggled into working-style continuity:

- `stable_vows`
- canonical governance posture
- one-shot `carry_forward`
- temporary emotions, frustration, or urgency
- a single council verdict
- one-off task tactics that have not repeated

If it changes what the system is *allowed* to do, it is not style.
If it only changes how an agent *usually proceeds*, it may belong here.

## Current Home In ToneSoul

ToneSoul does not need a new hot-state event lane for style.
The current bounded home is the latest `subject_snapshot`, especially:

- `decision_preferences`
- `verified_routines`
- `durable_boundaries` as guardrails, not as style payload

Packet/session-start readouts may derive a `working_style_anchor` from those fields.
That anchor is advisory only.

## Observability Before Promotion

Seeing a working-style anchor is not the same as knowing later agents are still following it.

ToneSoul may therefore derive a bounded `working_style_observability` summary from recent:

- `carry_forward`
- `next_actions`
- routing telemetry

Its purpose is narrow:

- show whether recent handoff surfaces still echo the shared style
- warn when style continuity is only partial or currently unreinforced
- reduce the temptation to pretend multiple agents already share one mind

This surface is descriptive only.
It does not certify obedience, alignment, or hidden cognition.
It only says whether recent shared residue still re-mentions the same operating habits.

## Receiver-Side Import Limits

Working style may be imported only as bounded workflow help.

Safe apply lanes:

- scan order
- evidence handling
- prompt shape
- session cadence
- render interpretation

Must-not-import lanes:

- vows or permissions
- canonical governance truth
- durable identity
- task scope or success criteria

If a later agent needs to change what the system is *allowed* to do, or what the current task *is*, working-style continuity must stop there.

## Fresh-Agent Validation

`session-start` may derive a bounded validation summary for fresh agents:

- is a shared working-style anchor visible?
- is a playbook visible?
- are import limits visible?
- is observability visible?
- is a non-promotion guard visible?

This validation does not prove that all agents now think the same way.
It only answers a narrower question:

`does the receiver have enough shared style structure to inherit bounded habits safely?`

## Receiver Rules

When a later agent sees working-style continuity:

1. `ack` is always safe
2. `apply` is allowed as a workflow habit
3. `promote` is not allowed without fresh evidence and deliberate review

In practice:

- safe to reuse:
  - scan order
  - prompt structure
  - verification cadence
  - render/encoding caveats
- not safe to promote automatically:
  - vows
  - durable identity claims
  - canonical planning truth
  - governance permissions

## Prompt-Discipline Relationship

Working-style continuity and prompt discipline are related but not identical.

- `TONESOUL_PROMPT_DISCIPLINE_SKELETON.md`
  - repo-wide prompt design discipline
- `TONESOUL_PROMPT_STARTER_CARDS.md`
  - practical prompt defaults any agent can reuse
- this contract
  - says how much of that style a later agent may inherit as bounded continuity

Prompt discipline is the repo's shared method.
Working-style continuity is the receiver-side question:

`how much of that method should I carry forward from the last agent?`

## Render And Encoding Relationship

Render/encoding habits also belong here in bounded form.

Example:

- correct habit: verify UTF-8 file content before calling `??` a corruption bug
- incorrect habit: assume every shell artifact is a file-layer problem

This contract therefore depends on:

- `TONESOUL_RENDER_LAYER_AND_ENCODING_BOUNDARY_CONTRACT.md`

## Runtime Readout Rule

If runtime exposes a `working_style_anchor`, it should be read as:

- `advisory`
- `apply if helpful`
- `do not promote`

Its job is to reduce needless style drift between agents, not to declare identity or law.

## Canonical Line

Shared R-memory can carry bounded operating style.
It cannot collapse multiple agents into one soul.
Use style continuity to preserve useful habits, not to pretend a shared mind already exists.
