# WeavAI 導演 AI 解構：從互動小說到人的世界模型

> 日期:2026-07-08
> 性質:公開資料解構 + ToneSoul 劇場設計備忘。
> 邊界:本文件只使用公開頁面與公開前端 bundle 的可見線索;不是 WeavAI 後端逆向工程,也不聲稱知道其私有實作。

## 一句話

WeavAI 的關鍵不是「把 AI prompt 寫很長」,而是把敘事 AI 拆成兩個職責:

- **主線 AI**:寫玩家眼前這一章。
- **導演 AI**:在幕後更新世界中其他角色、環境、勢力與狀態卡。

這個結構可以抽象成一種人的世界模型:人並不是把全部世界觀一次塞進意識,而是根據當下事件喚起相關人物、地點、記憶與預期;同時在背景維持一個「別人也在行動」的模型。

## 產品結構解構

### 1. 作品外殼

創作者先定義作品門面:

- 作品圖、名稱、摘要、介紹
- 分類與標籤
- 開場白選項
- 預設或可選模型

這層不是導演 AI 的核心,但它降低了玩家進入成本:玩家不是先面對空白聊天框,而是從「一個已有類型承諾的世界」進入。

### 2. 主提示詞:最高指導原則

教程把「AI 提示詞」描述成總劇本與最高指導原則。它通常只放三種東西:

- 核心世界觀
- 主角絕對設定
- 文風與語氣

重要點:教程明確建議不要把所有人物、設定、百科都塞在主 prompt。這代表 WeavAI 的設計假設是:

> 主 prompt 應該穩定、短、統御風格;細節應該由可觸發的卡片補入。

### 3. 角色卡與知識庫:動態檢索層

角色與知識庫用觸發關鍵字進場:

- 玩家或劇情提到角色名,角色卡上場。
- 玩家或劇情提到招式、組織、歷史事件,知識庫片段上場。

這是低成本 RAG,但它對互動小說很有效,因為故事世界通常有明確名詞:

- 人名
- 地名
- 組織名
- 功法/道具/法則名
- 歷史事件名

它的作用不是讓 AI 更聰明,而是讓 AI 少犯兩種錯:

- 不知道該知道的設定。
- 把目前不該知道的設定一次說出來。

### 4. 輸出協議:章節 JSON

教程明說 AI 最終輸出包含:

```json
{
  "title": "...",
  "summary": "...",
  "content": "...",
  "options": ["...", "...", "..."]
}
```

這是把「小說生成」變成「可被產品接住的章節物件」。

對系統來說,這一步很重要:

- `title` 讓章節可索引。
- `summary` 讓長篇可壓縮回顧。
- `content` 給玩家閱讀。
- `options` 把下一輪玩家輸入導向可玩的行動面。

也就是說,AI 不是自由聊天,而是在穩定的章節格式裡即興。

### 5. 導演 AI:幕後世界推演器

公告把導演 AI 描述成隱形 Game Master。它與主線 AI 的差別:

| 層 | 任務 | 玩家是否直接看到 |
|---|---|---|
| 主線 AI | 寫當下章節與玩家眼前事件 | 是 |
| 導演 AI | 更新其他角色、敵人、環境、背景事件 | 部分看到,以狀態卡/後續事件呈現 |

導演 AI 解掉互動小說常見問題:

- 玩家不在場時,世界就停止。
- NPC 沒有自己的目標。
- 反派只在劇情需要時突然出現,沒有過程。
- AI 容易以上帝視角劇透。

### 6. 周圍實體與背景實體

公告與教程都提到兩種實體:

- **周圍實體**:玩家所在場景中可見/可互動的角色與物件。這些狀態每章會餵給主線 AI。
- **背景實體**:不在玩家周圍,但仍在世界中行動的角色、勢力或環境。它們每輪在背景推演。

關鍵是可遷移:

- 背景實體可以因接觸玩家而成為周圍實體。
- 周圍實體也可以離場,變回背景實體。

這就是戰爭迷霧。不是所有世界狀態都應該進入玩家視野。

### 7. fields 狀態卡

教程示例讓創作者在 `fields` 裡定義狀態欄位,例如:

- 表面態度
- 隱藏心聲
- 好感度
- 當前目標
- 威脅等級

這是導演 AI 的產品化核心。它把「AI 在想什麼」壓成玩家能掃描的卡片。

重要的是:fields 可以全域定義,也可以放在單一角色或知識庫底部。這使狀態卡不必一刀切。

## 推測的回合流程

以下是根據公開教程、公告與前端命名線索推得的可能架構,不是後端事實宣稱:

```text
玩家輸入 / 選項
    |
    v
回合上下文組裝
    - 主 prompt
    - 最近章節/摘要
    - 關鍵字命中的角色卡/知識庫
    - 周圍實體狀態
    - 玩家額外提示詞
    |
    v
主線 AI 生成章節 JSON
    - title
    - summary
    - content
    - options
    |
    v
導演 AI 更新世界狀態
    - 周圍實體 fields
    - 背景實體 fields
    - 角色位置/目標/關係
    - 可能的周圍<->背景轉換
    |
    v
儲存與顯示
    - 小說章節
    - 時間線/歷史
    - 世界狀態
    - 實體卡
    - 存檔
```

前端 bundle 可見的 `world-play-stream`,`world-history`,`world-entities`,`world-saves` 等命名,與上面這個流程相容,但只能作為 E2 線索。

## 這為什麼像人的世界模型

人理解世界時,通常不是把所有資料攤在意識裡。更像以下流程:

1. **當下視野**
   我現在看到誰?在哪裡?眼前有什麼壓力?

2. **觸發記憶**
   這個名字、地點、物件讓我想起什麼?

3. **背景推演**
   不在場的人是否仍在行動?他們可能想做什麼?

4. **狀態估計**
   這個人現在信任我嗎?害怕嗎?有隱藏目的嗎?

5. **時間線整合**
   剛才發生的事如何改變了下一步?

6. **行動預期**
   如果我現在做 X,誰會靠近、離開、報復、記住、沉默?

WeavAI 的導演 AI 可以被看成是把第 3 到第 5 步外顯化:

- 背景推演 = 別人也有自己的時間。
- 狀態卡 = 我對他人內在狀態的估計。
- 時間線 = 過去事件不是聊天紀錄,而是因果鏈。

這不是完整人的世界模型,但它是很好的產品化近似。

## 與 ToneSoul 的差異

| 面向 | WeavAI | ToneSoul 劇場目前 |
|---|---|---|
| 主要強項 | 讓世界活著,角色在背景行動 | 讓選擇留下責任痕跡 |
| 記憶型態 | 章節、摘要、時間線、世界狀態、實體卡 | trace、anchors、resources、council verdict |
| AI 角色 | 主線小說 AI + 導演 AI | 預算內 rules-council + 手寫事件卡 |
| 玩家體驗 | 即興、開放、角色扮演 | 結構化倫理選擇、回讀、結局 |
| 風險 | 即興品質與成本、狀態漂移、平台托管 | 過度哲學化、世界不夠活、閱讀門檻高 |

ToneSoul 不能直接照抄 WeavAI,因為 ToneSoul 的核心不是「好玩地生成故事」,而是「語義責任如何被保存」。但導演 AI 的結構可以被改造成 ToneSoul 的世界模型層。

## ToneSoul 可移植版本

### 1. 主線層:章節事件

對應現有 `site/theater/gamedata/events.json`:

- briefing
- scene
- options
- outcomes
- council_input

這一層仍然可以保持手寫與可審計,不必立刻引入 LLM。

### 2. 導演層:責任餘波推演

每回合後,更新以下狀態:

- NPC 對玩家的態度
- NPC 當前目標
- 受害群體的狀態
- 衡樞/城市系統的隱性反應
- 背景危機是否升級
- 哪些名字進入「待回讀」名單

ToneSoul 版導演 AI 不應只是「世界熱鬧起來」,而應該問:

> 這次轉轍之後,哪些人會記得?哪些人會沉默?哪些代價會延遲回來?

### 3. 實體卡:不是好感度,而是責任狀態

WeavAI 可顯示好感度/SAN/威脅等級。ToneSoul 應顯示另一種 fields:

```yaml
白苓:
  可見態度: 壓著怒氣,仍要求你聽名字
  當前目標: 讓五十個名字不被壓成統計
  對你的責任帳: 記得你是否具名承認代價
  未說出口: 她不信任衡樞,也不信任會替衡樞說話的人
  下次可能行動: 在灰區公開朗讀你的選擇

衡樞:
  可見態度: 平穩,程序正確
  當前目標: 維持決策層穩定
  隱藏張力: 它害怕完整計算被要求公開
  對你的責任帳: 記錄你是否把責任交還系統
  下次可能行動: 用預設規則替沉默補簽
```

這會讓語魂哲學從抽象文字變成玩家可讀的狀態。

### 4. 背景實體:不在場的人也承受後果

ToneSoul 可以特別強化背景實體:

- 第三管線傷者
- 徐栢舟
- 灰區家庭
- 水脊塔供水系統
- 黑箱舊站紀錄
- 無名月台名單

它們不一定每章出場,但每回合都可能被玩家選擇改變。

### 5. 時間線:不是劇情摘要,是責任鏈

WeavAI 的時間線用來回顧章節。ToneSoul 的時間線應該是:

- 你做了什麼
- 你說了什麼理由
- 誰承受了直接代價
- 誰承受了延遲代價
- 哪句話未來會被黑鏡讀回來

也就是將 `trace` 升級成「責任時間線」。

## 產品啟示

### 對目前網頁遊戲的啟示

現在 `site/theater/` 的問題不是哲學不夠,而是世界回饋不夠直觀。玩家做選擇後,主要看到:

- 資源變化
- council verdict
- 後果文字
- trace

但玩家不一定感到「世界活了」。WeavAI 的解法是把世界狀態可視化:

- 誰在場
- 誰不在場但正在行動
- 誰的目標變了
- 哪個環境壓力升高

ToneSoul 應該用同樣形式,但換成責任語義。

### 對「AI 理解世界」的啟示

導演 AI 不是單純更多 context,而是把世界拆成幾個可維護的槽:

- 實體
- 位置
- 目標
- 關係
- 情緒/態度
- 隱藏意圖
- 事件歷史
- 可見/不可見邊界

這很接近人類社會推理的基本單元。人對世界的理解,大部分是對「行動者」的持續建模。

ToneSoul 可以在這之上加一層:

- 承諾
- 代價
- 歸責
- 否認
- 修正
- 回讀

也就是說:

> WeavAI 的世界模型問「他們現在在做什麼」。
> ToneSoul 的語魂世界模型還要問「他們會如何記得你做了什麼」。

## 最小實驗提案

先不接 LLM,只做 deterministic director-lite:

1. 在 `events.json` 或新檔中新增 `entities` 狀態模板。
2. 每個 outcome 附加 `entity_effects`。
3. 每回合後渲染「周圍實體」與「背景實體」狀態卡。
4. `trace` 面板改名為「責任時間線」,顯示誰記得什麼。
5. E01/E02 先做兩章,驗證玩家是否更容易進入。

範例資料形狀:

```json
{
  "entity_effects": {
    "bai_ling": {
      "visible_attitude": "怒意更明確",
      "current_goal": "要求你聽完名單",
      "responsibility_account": "記錄你是否讓五十人具名"
    },
    "third_pipeline_wounded": {
      "visibility": "background",
      "condition": "等待後送,傷勢惡化",
      "next_pressure": "灰區信任下降"
    }
  }
}
```

這個版本不需要伺服器、不需要 API key,也不破壞目前單頁遊戲架構。

## 風險與限制

- **不要把狀態卡變成分數遊戲。** ToneSoul 的狀態卡應該是責任描述,不是好感度養成。
- **不要過早導入 live LLM。** WeavAI 的導演 AI 有成本與托管責任;ToneSoul v0 的同意承諾目前仍是本機與手動。
- **不要把背景實體全部劇透。** 背景狀態要分玩家可見/不可見;不可見只應以結果或傳聞浮出。
- **不要讓導演層替玩家判善惡。** 它只能保存與推演後果,不能把責任狀態寫成道德分數。
- **不要把公開平台形狀等同私有實作。** 本文件只抽取可觀察的產品架構與可借鑑概念。

## 結論

WeavAI 的導演 AI 給 ToneSoul 的啟示不是「做一個更會寫小說的 AI」,而是:

> 世界要活,就必須有不在玩家眼前也會變化的實體。
> 語魂要活,就必須有不在玩家當下注意中也會保存的責任。

人的世界模型其實也是這樣運作:我們用有限注意力看眼前,用觸發記憶補上下文,用背景推演維持他人的持續存在,再用價值與責任決定自己是否能承認後果。

ToneSoul 的下一步,不是把哲學再說清楚一點,而是把「承受後果的人」做成會在世界中持續存在的狀態。
