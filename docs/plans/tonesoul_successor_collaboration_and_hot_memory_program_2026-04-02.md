# ToneSoul Successor Collaboration And Hot-Memory Program (2026-04-02)

> Purpose: turn the current continuity stack into a successor-safe collaboration system where any later AI can continue work from repo-native surfaces instead of chat-history guesswork.
> Status: approved planning program, not yet the only active short board
> Authority: planning aid. Does not outrank `task.md`, runtime code, tests, or canonical contracts.

---

## Goal

Make ToneSoul good enough that:

- a fresh AI can enter through the normal stack
- reconstruct the current center of gravity
- understand what is stable, contested, and stale
- know what may be applied versus only acknowledged
- continue unfinished work without inventing a parallel architecture story

## Why This Program Exists

ToneSoul already has many strong pieces.
What it still lacks is cleaner successor coherence across those pieces.

Current symptoms:

- entry surfaces still drift in emphasis
- successors can read the same packet and still reconstruct different "main stories"
- chat history still explains too much of the true center
- hot-memory is richer than before, but its layers are not yet obvious enough to a cold reader
- summary children are still at risk of being over-trusted

## The North Star

A successor AI should be able to answer, within 10 minutes:

1. What is the current launch truth?
2. What is the current short board?
3. What is stable?
4. What is contested?
5. What is stale?
6. What is advisory only?
7. Which bounded next move is safest?

If it cannot do that, the hot-memory system is still too implicit.

## Workstream 1: Successor Entry Convergence

Problem:
the first five minutes still depend too much on which surface the successor happens to open first.

Deliverables:

- clean repo-local AI entry surfaces
- one stable successor reading order
- one "first 10 minutes" contract

Concrete moves:

1. keep `AI_ONBOARDING`, `AI_QUICKSTART`, `DESIGN`, `start_agent_session`, and `observer_window` aligned
2. ensure GitHub/Copilot style agents do not start from stale or mojibake-heavy surfaces
3. expose current tier, short board, and do-not-promote rules in the first-hop bundle

Exit criteria:

- a low-context successor can enter without opening long historical docs first

## Workstream 2: Low-Drift Anchor Phase 2

Problem:
observer window exists, but the anchor center is still more descriptive than directive.

Deliverables:

- stronger low-drift anchor selection rules
- explicit stable/contested/stale source precedence
- reverse-check against known anchor drift before successor continuation

Concrete moves:

1. define which surfaces can feed the anchor directly
2. downgrade repeated carry-forward without fresh evidence
3. expose anchor conflicts instead of silently smoothing them

Exit criteria:

- successors can see the current center without mistaking summaries for law

## Workstream 3: Hot-Memory Ladder And Compaction Hygiene

Problem:
ToneSoul has many continuity surfaces, but their ladder is still clearer to implementers than to successors.

Deliverables:

- explicit hot-memory ladder
- compaction / checkpoint / packet / observer responsibilities
- quarantine lane for low-confidence residue

Concrete moves:

1. define which state belongs in anchor, coordination, handoff, working identity, replay, and residue lanes
2. ensure compaction preserves open questions and blockers instead of only smooth summaries
3. separate resumability from governance and identity more aggressively

Exit criteria:

- successors stop over-reading compaction and under-reading canonical center

## Workstream 4: Subsystem Parity And Gap Truth

Problem:
ToneSoul has many strong subsystems, but no one current document says which are baseline-complete, partial, deferred, or overclaim-prone.

Deliverables:

- a ToneSoul-native parity and gap map
- status terms that are honest enough for collaborator-beta use

Concrete moves:

1. map subsystem maturity using `baseline / partial / deferred / beta-usable`
2. attach each subsystem to its strongest evidence and next bounded move
3. explicitly name the overclaim that must be avoided for each lane

Exit criteria:

- successors stop treating every subsystem as equally mature

## Workstream 5: Anti-Fake-Completion And Closeout Grammar

Problem:
later agents can still inherit a clean story when the real state is contested, underdetermined, or blocked.

Deliverables:

- closeout grammar that preserves unresolved items
- stronger fail-stop / underdetermined handoff posture
- open-questions lane in bounded resumability surfaces

Concrete moves:

1. require handoff artifacts to name unresolved blockers explicitly
2. keep descriptive-only and contested states visible in closeout
3. refuse false completion when evidence is thin

Exit criteria:

- no successor artifact can look "done" while hiding active uncertainty

## Workstream 6: Hooks, Permission Chain, And Mutation Preflight

Problem:
continuity is stronger than before, but mutation discipline still relies too much on operator skill and scattered rules.

Deliverables:

- a ToneSoul-native write-preflight / mutation-hook map
- bounded guard chain for:
  - memory writes
  - task-board mutation
  - launch-facing claim language
  - publish / push posture

Concrete moves:

1. map the current write/mutate/publish decision points
2. classify which ones need hooks, which only need readout posture, and which should remain human-gated
3. keep the chain explicit rather than magical

Exit criteria:

- successors can tell where mutation discipline actually lives
