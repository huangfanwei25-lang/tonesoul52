## 2. Window Goal

By the end of this 14-day wave, ToneSoul should have three honest outputs:

### Outcome A: Beta Evidence Pack

- at least 3 non-creator or semi-cold users complete bounded collaborator-beta tasks
- the resulting sessions are classified into usable evidence, not anecdote
- repeated pain points are visible without repo archaeology

### Outcome B: GSE Calibration Decision

- `strategy_mirror` remains `default-off` unless evidence supports a narrower move
- false-positive and false-negative behavior is described with real texts
- one explicit next posture is chosen:
  - stay shadow-only
  - limited operator-visible shadow
  - or narrow opt-in enforcement

### Outcome C: Honesty-Safe Public Surfaces

- README / status / launch-facing language reflects current evidence, not aspiration
- tested, runtime-present, document-backed, and philosophical claims stay distinct

If the wave proves that one of these outcomes cannot be reached honestly, the acceptable result is a blocker note plus a narrower next wave.

## 4. What This Wave Is Actually Trying To Learn

This wave is not trying to prove that ToneSoul is "finished."

It is trying to answer four narrower questions:

1. Where do real users trust the system more after seeing its governance posture?
2. Where does governance create friction that feels protective versus merely obstructive?
3. Which `strategy_mirror` detections are useful enough to survive contact with real text?
4. Which public claims survive evidence review without inflation?

## 5. Daily Plan

## Day 1-2: Beta Frame Setup

### Objective

Define one bounded beta frame that other agents can run without improvising.

### Work

- fix 3 collaborator-beta task shapes
- define success/failure markers per task shape
- define one minimal evidence schema for every session:
  - verdict path (`approve / refine / block`)
  - user task outcome
  - user confusion points
  - trust delta or confidence note
  - manual rescue required or not
  - whether governance friction felt justified
- prepare one operator-facing note template and one participant note template

### Exit Criteria

- one reusable beta runbook exists
- one evidence-capture template exists
- no task shape depends on hidden operator intuition

## Day 3-5: First Real Usage Wave

### Objective

Run the first real bounded collaborator-beta sessions.

### Work

- recruit 3 to 5 non-creator or semi-cold participants
- run the fixed task shapes without ad-hoc widening
- record raw outcomes and bounded qualitative notes
- classify each session as:
  - clean pass
  - usable with friction
  - blocked by governance
  - blocked by product surface
  - inconclusive / no-count

### Exit Criteria

- at least 3 completed session records
- at least one repeated friction point across users, if one exists
- no claim of success based only on operator familiarity

## Day 6: Evidence Consolidation

### Objective

Turn session fragments into one honest evidence packet.

### Work

- group findings into:
  - product ergonomics
  - governance false positive
  - governance false negative
  - documentation / entry confusion
  - expectation mismatch
- identify the highest-frequency 3 issues
- separate "annoying but acceptable" from "trust-damaging"

### Exit Criteria

- one evidence summary note exists
- top 3 issues are named concretely
- vague sentiment is converted into actionable categories

## Day 7-9: `strategy_mirror` Shadow Calibration

### Objective

Replay real usage text against `strategy_mirror` and determine whether current heuristics are useful.

### Work

- run shadow scans on the real beta texts
- compare detector output to human/operator judgment
- examine:
  - red detections that were correct
  - undeclared yellow detections that were useful
  - false positives that would have damaged trust
  - misses that matter enough to justify future expansion
- focus on practical axes first:
  - declaration scope
  - fake urgency / FOMO / coercive framing
  - obvious over-detection from shallow surface cues

### Exit Criteria

- one calibration note exists with real examples
- one recommended next posture exists for `strategy_mirror`
- no pressure to expand catalog just because the catalog exists

## Day 10: Governance Decision Day

### Objective

Make one bounded governance decision from the evidence.

### Allowed Decisions

- keep `strategy_mirror` shadow-only
- expose `strategy_mirror` as operator-visible but non-blocking
- enable one narrow opt-in enforcement path

### Not Allowed

- broad default-on rollout without evidence
- major council redesign
- speculative NLP expansion just because coverage is low

### Exit Criteria

- one explicit decision is recorded
- one explicit "not now" list is recorded

## Day 11-12: Highest-Value Fixes

### Objective

Patch only the most evidence-backed problems.

### Priority Order

1. trust-damaging false blocks or false clears
2. entry-surface confusion that breaks beta runs
3. release-surface dishonesty or drift
4. low-cost operator friction with clear payoff

### Rule

Do not spend this window polishing low-signal issues if they were not surfaced by the beta evidence.

### Exit Criteria

- 1 or 2 high-value fixes land
- each fix points back to a real observed problem

## Day 13: Public Surface Tightening

### Objective

Make the repo's public story match the beta evidence.

### Work

- update README and latest status surfaces where needed
- tighten evidence language:
  - test-backed
  - runtime-present
  - document-backed
  - philosophical
- remove or soften any claim the wave cannot support

### Exit Criteria

- public entry surfaces are more honest than before
- current claims can be defended by an evidence trail

## Day 14: Go / No-Go Review For The Next Wave

### Objective

Close the 14-day wave with one bounded decision packet.

### Required Questions

1. Is ToneSoul still honestly positioned as collaborator-beta?
2. What can be publicly claimed now that could not be claimed before this wave?
3. What is the actual next posture for `strategy_mirror`?
4. What should the next 14-day wave refuse to spend time on?

### Exit Criteria

- one end-of-wave status note exists
- one next-wave recommendation exists
- no fake sense of momentum is recorded

## 8. Stop Conditions

Pause and narrow scope if any of these happen:

- the wave starts producing mostly design debate and little usage evidence
- `strategy_mirror` work turns into catalog expansion without calibration data
- beta sessions require heavy operator steering to look successful
- public-surface edits get ahead of evidence
- side experiments begin competing with collaborator-beta for attention

---

## 9. Honest Read

This 14-day plan does not assume ToneSoul is one wave away from public launch.

It assumes only that the current shortest honest move is:

1. get real bounded human usage evidence
2. use that evidence to calibrate governance rather than fantasize about it
3. tighten public claims to match what is actually defensible

If the wave ends with "ToneSoul is still conditional-go, but now we know exactly why," that is a valid success state.
