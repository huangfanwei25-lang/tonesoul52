# ToneSoul Consistency-First Governance Depth Program (2026-04-07)

> Purpose: queue the next bounded program after the current self-improvement short board so ToneSoul can become faster on low-risk turns, more honest on high-risk turns, and more consistent across Codex / Claude-style / dashboard consumers.
> Status: accepted follow-through program candidate, queued behind the current self-improvement short board.
> Authority: planning aid. Runtime truth remains in code, tests, and accepted architecture contracts.

---

## Why This Program Exists

ToneSoul's real current user is still:

`the next AI that has to load the repo and continue safely`

That means today's core problem is not public polish.
It is:

- cross-agent consistency
- first-hop discoverability
- honest high-risk handling
- reduced latency without deleting governance

Recent work showed a real gap:

- another AI still missed `Take E` / hot-memory compression even though the docs already existed
- the repo can carry the same hot state, but different consumers still assemble different stories

At the same time, external research pressure sharpened another truth:

- not every turn deserves full governance cost
- but high-risk turns deserve more honesty, not less

This program exists to combine those two pressures into one bounded next bucket.

---

## Why This Is A New Line

This is not just more of the self-improvement loop.

The self-improvement loop asks:

- how may ToneSoul improve bounded operator/runtime surfaces without drifting into selfhood or governance inflation?

This new line asks:

`how should ToneSoul route governance cost, keep cross-agent interpretation aligned, and earn the right to future high-risk acceptance tests?`

So this line is about:

- routing depth
- grounding burden
- fail-stop discipline
- cross-agent acceptance

not about broader shell polishing.

---

## Compressed Thesis

ToneSoul should:

1. keep multi-agent and governance structure
2. stop paying full-governance cost on every low-risk turn
3. strengthen high-risk honesty before claiming domain readiness
4. treat cross-agent consistency as the immediate acceptance target

In short:

`faster where safe, stricter where risky, and more consistent everywhere.`

---

## What This Program Is Not

This program is **not**:

- a rewrite into a fresh `Module 0 / A / B / C` universe
- a claim that ToneSoul is now a hardware-level neuro-symbolic system
- permission to bypass reflex, vows, or output gates
- permission to claim stock-analysis readiness today

It is a bounded follow-through program built on current ToneSoul seams.

## Proposed Phase Ladder

### Phase 848: First-Hop Compression Boundary Refresh

Bounded scope:

- make `Take E` / hot-memory compression easier to find from AI-first-hop surfaces
- keep the change inside onboarding, quickstart, and current handoff
- avoid reopening the whole successor/hot-memory theory bucket

Success criteria:

- a fresh later agent no longer needs deep prose luck to notice the hot-memory decay/compression map when compaction or resumability is in scope

### Phase 849: Governance Depth Routing v0 Spec

Bounded scope:

- formalize `light / standard / full`
- keep routing orthogonal to model/path routing
- explicitly preserve reflex, vow, and output-edge posture

Success criteria:

- ToneSoul has one formal shared story for earned governance cost that later agents can implement without importing a foreign module universe

### Phase 850: Governance-Depth Runtime Pilot Design

Bounded scope:

- identify the smallest live seam for `governance_depth`
- keep current default behavior safe and backward compatible
- decide which heavy subpaths are actually skippable under `light`

Success criteria:

- later implementation can proceed from one explicit seam instead of hand-wavy "fast path" intuition

### Phase 851: High-Risk Grounding Check Design

Bounded scope:

- define a bounded post-hoc claim grounding pass
- activate only for verify-heavy / fact-heavy / high-risk turns
- avoid pretending ToneSoul can do token-level reverse pointers

Success criteria:

- ToneSoul has one honest design for tracing core claims back to visible support before calling a high-risk answer acceptable

### Phase 852: Verification Over-Budget Fail-Stop Design

Bounded scope:

- define when repeated revision / failed grounding / verification cost should stop the loop
- preserve fail-stop honesty and anti-fake-completion grammar
- keep the stop reason visible and actionable

Success criteria:

- ToneSoul can explain when to stop verifying and ask for better support instead of spending more compute theatre

### Phase 853: Cross-Agent Consistency Acceptance Wave

Bounded scope:

- run the same bounded task through at least two consumer shells
- compare:
  - first-hop ordering
  - closeout reading
  - compression-boundary reading
  - governance-depth recommendation
- judge consistency, not literary similarity

Success criteria:

- Codex-style and Claude-style consumers reach the same bounded continuation story often enough that later domain testing becomes meaningful

### Phase 854: Deferred High-Risk Domain Trial Gate

Bounded scope:

- define the conditions under which a future stock-analysis or similar high-risk task may be used as an acceptance test
- keep this gated behind grounding and fail-stop readiness
- make overclaim language explicit

Success criteria:

- ToneSoul has an honest rule for when a future high-risk domain trial is allowed, instead of treating interesting demos as proof of readiness
