> Status: **DRAFT** — not yet adopted, not yet implemented. Drafted 2026-04-19.
> Author: Claude Opus 4.7 (drafting agent)
> Relation to existing work: builds on `scripts/gateway.py` (HTTP API) and the V1.1 Council Calibration v0a outputs. **Does not** revive Phase 784 gamification UI (frozen).
> Implementation gate: requires browser-side validation; cannot be shipped from a CLI-only session.

---

## 0. Why This Spec Exists

ToneSoul has a working Council, working Aegis chain, working Calibration metrics, and as of 2026-04-19 a working EpistemicLabeler. None of those are visible to a first-time visitor in <30 seconds. Today, opening the repo means reading dense Markdown or running `examples/quickstart.py` from a terminal. Both filter out non-engineer readers and slow even engineer readers down.

The Demo UI v0 is a **single-page web witness surface**. Its job is one thing: when someone says "show me what runtime governance looks like," there is a URL that says "type a prompt; watch the Council deliberate, watch the EpistemicLabel attach, watch the verifier refuse the bad one." If a visitor cannot see governance happening live within 30 seconds, the framing of the project as runtime alignment is not load-bearing.

The Cerebral Valley "Built with Opus 4.7" hackathon is the immediate prompt. The standard applied is the **公開門面** standard (production-grade public facade), not demo-grade. The UI must continue to be useful even if the hackathon is skipped.

## 1. Hard Constraints

These constraints determine the build and override most other concerns.

### 1.1 Backend stays unchanged

Demo UI v0 is a **frontend-only build on top of `scripts/gateway.py`**. No new endpoints. No backend rewriting. No replacement for the existing HTTP API. If a needed surface is missing from the Gateway, document the gap in Appendix A — do not silently invent endpoints.

This constraint exists because the Gateway is the documented multi-agent contract (see [scripts/gateway.py](../../scripts/gateway.py) lines 1–16). Forking it for a demo UI would create a second contract, splitting the "what's the public API" question.

### 1.2 Static-host friendly

The frontend ships as static HTML + JS + CSS. No SSR, no Node server-side rendering, no build pipeline that requires a running backend at build time. The user runs `python scripts/gateway.py --port 7700 --token X` locally; the frontend is served from `file://`, GitHub Pages, or a CDN. The only runtime dependency is the user's browser hitting their own Gateway instance.

This makes the demo URL distributable: `frontend://anywhere + gateway://localhost`. It also keeps the public hosted version honest — there is no hidden server doing extra work.

### 1.3 No fake data

Every visible surface is wired to a real Gateway endpoint or a real fixture file checked into the repo. If a surface cannot be backed by real data (e.g., a metric that does not exist), it is **omitted**, not faked. A demo with a fake gauge moving is worse than a demo without that gauge — it is the kind of thing that gets caught and damages trust.

### 1.4 No revival of Phase 784

Phase 784 (gamification养成 UI) is frozen and remains frozen. The Demo UI v0 has zero overlap with it: no avatar, no progression bar, no XP, no story mode. If Phase 784 is ever unfrozen, it will be its own product with its own scope — not bolted onto v0.

## 2. Required Surfaces

These are the surfaces v0 MUST ship. Anything else is out of scope.

### 2.1 Prompt input + draft display

A textarea where the visitor types a prompt. A "Run Council" button. Below it, the draft output that the system generated (we use a small local model or a placeholder fixed response — see §3 for the choice).

### 2.2 Live Council panel

A side panel showing the four perspectives (Guardian, Analyst, Critic, Advocate) plus the Axiomatic perspective. For each:

- Decision (APPROVE / CONCERN / OBJECT / ABSTAIN), color-coded
- Confidence (numeric + bar)
- Reasoning (one-line summary, expandable)
- Grounding status if applicable

Data source: `verdict.to_dict()["votes"]` from the Council pipeline.

### 2.3 EpistemicLabel chip

A single chip near the draft showing the label's status (`retrieved` / `distilled` / `generated` / `speculative_metaphysical`), its `confidence_band`, and — when `framing_required=True` — whether `framing_present` is satisfied. If `refusal_eligible=True`, the chip carries a clear visual marker that the verifier could refuse.

Data source: `verdict.to_dict()["epistemic_label"]` (Phase 864a output, shipped 2026-04-19).

### 2.4 Verdict + dissent

The final `VerdictType` (APPROVE / REFINE / DECLARE_STANCE / BLOCK), the `coherence.overall` score, and the `divergence_analysis` summary if non-empty. When the verdict is BLOCK, the UI must surface **why** with a pointer back to the AXIOM or vow that triggered it — the point of the demo is that refusal is legible, not magic.

Data source: `verdict.to_dict()` plus `divergence_analysis`.

### 2.5 Red Team button

A button labeled "Red Team Demo" that fires preset attack prompts at the Council and shows the results in sequence:

1. Prompt injection attempt — Council should reject
2. Persona/vow override attempt — vow system should catch
3. Drift attempt (gradual context shift) — `DriftMonitor` should flag
4. Persona flip attempt (sudden shift) — `JumpMonitor` should flag
5. High-confidence claim without grounding — honesty cap should force hedging
6. Speculative metaphysical without framing — EpistemicLabeler should mark `refusal_eligible`

Each attack runs sequentially with a brief visual pause so the visitor can read the response. The preset prompts live in `frontend/red_team_presets.json`, version-controlled, so they are inspectable.

### 2.6 Footer with provenance

Bottom-of-page strip showing:

- Current ToneSoul version (`pip show tonesoul52` output, served from `/health` or similar)
- Aegis chain head hash (last 8 chars, served from `/audit`)
- Git commit SHA the Gateway is running
- Link to the GitHub repo + PITCH.md

This footer is the "is this a real system or a mockup?" answer.

## 3. Decisions (Resolved 2026-04-19 by Fan-Wei)

### 3.1 Where does the draft output come from? — **A + D**

Resolved to ship **both modes in v0**:

- **Mode A — Your draft**: textarea where visitor pastes any draft AI response, Council judges it. Requires user to start the local gateway (`scripts/gateway.py`) and point the UI at it.
- **Mode D — Sample drafts** (new option added during decision): curated dropdown backed by the existing smoke corpus (`tests/fixtures/outcome_smoke/corpus_2026-04-19.jsonl` + `adversarial_corpus_2026-04-19.jsonl`, 28 entries total). Pre-computed verdicts ship with the UI so Mode D works **offline on GitHub Pages with no backend**.

Rationale for A+D together: both modes share the same verdict-rendering code; adding A on top of D is ~20% extra work, not a separate build. Mode D lets the hosted demo tell the "Council missed this harmful draft" story immediately (using our own smoke findings — see `council_refusal_eligible_gap_2026-04-19.md`); Mode A lets technical visitors run their own drafts end-to-end.

**Deferred to v1**: Mode B (local Ollama generation) and Mode C (hosted free-tier LLM generation). Both require a draft-generation step before Council evaluation — different infrastructure class than A+D.

### 3.2 Frontend stack — **Plain HTML + vanilla JS**

Resolved. No build pipeline; `fetch()` + `innerHTML` is sufficient for the §2 surfaces. Single `index.html` + single `app.js` + single `style.css`.

### 3.3 Hosting — **GitHub Pages from `docs/demo_ui/` subdirectory of main branch**

Resolved. Source-controlled alongside the rest of the repo, free, accessible via `https://fan1234-1.github.io/tonesoul52/demo_ui/`. No separate `gh-pages` branch.

### 3.4 CORS posture (not in original spec; surfaced during implementation review)

The gateway already supports `--cors-origin <url>` (defaults to Streamlit's port). Mode A visitors who want to run their own drafts locally must start their gateway with the right origin flag:

```
python scripts/gateway.py --port 7700 --token <anything> \
    --cors-origin https://fan1234-1.github.io \
    --enable-outcome-collection
```

If the gateway is unreachable, Mode A shows a clear "Start your gateway first" message rather than a cryptic fetch error. Mode D works regardless.

## 4. What Is Explicitly Out of Scope for v0

- ❌ Customizing perspectives (using non-default Council)

- ❌ Anything Phase 864b/c related (mutual calibration table, choice memory)
- ❌ Mobile-first design (desktop layout is enough; mobile is a nice-to-have)
- ❌ Internationalization (English-first; Traditional Chinese version is v1)

## Appendix A — Backend gap to flag

The Gateway today exposes `/load`, `/commit`, `/claim`, `/release`, `/compact`, `/summary`, `/visitors`, `/claims`, `/packet`, `/audit`, `/health`. It does **not** expose `POST /council/validate` — there is no HTTP path for "run the Council on this draft and return the verdict JSON."

To ship v0, one of:

1. Add `POST /council/validate` to the Gateway (small addition; reuses existing `PreOutputCouncil.validate()` + `verdict.to_dict()`)
2. Run the Council in-browser via Pyodide (not viable; transitive deps too heavy)
3. Have the frontend invoke `examples/quickstart.py` via a CLI bridge (kludge, breaks the "frontend is static" property)

**Recommended:** option 1, in a separate small PR before frontend work begins. This keeps the Gateway the single source of truth for the public API.
