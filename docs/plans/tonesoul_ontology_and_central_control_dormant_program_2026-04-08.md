## 1. Why This Program Exists

The ontology / central-control rethink surfaced a real pressure:

- `unified_pipeline.py` currently carries both driving-adapter behavior and domain-logic pressure
- the system has grown enough semantics that "what is core?" is no longer obvious
- memory, plan, and knowledge surfaces risk multiplying faster than they are being clarified

This pressure is real.

It should not be denied.

But it should also not be confused with the current shortest board.

Today ToneSoul has just crossed into collaborator beta with a `CONDITIONAL GO`.
That means the immediate responsibility is:

- stable operations
- repeated e2e evidence
- honest launch boundaries

not:

- a large architectural extraction
- a new central-control regime
- a fresh semantic universe during beta stabilization

This program exists so the pressure stays visible without hijacking the active lane.

## 2. Core Thesis

ToneSoul likely does need a smaller and more explicit center over time.

That center should probably look like:

- a bounded Domain Core
- explicit ports for memory / deliberation / audit
- clearer distinction between advisory compilation and canonical governance truth

But the right near-term move is:

`name the pressure, bound the questions, and delay activation until the beta lane is no longer fragile`

## 3. Why This Is Dormant Instead Of Active

### Reason 1: Beta just became conditionally viable

Phase 726 did not say "rewrite now."

It said:

- collaborator beta is honest enough to proceed
- the main remaining risk is evidence thickness, not missing ontology

### Reason 2: The changes touch core seams

Each major idea in the ontology file reaches into high-risk surfaces:

- `Domain Core` extraction touches runtime boundaries
- `Dormancy` touches task / memory / capability state semantics
- `Dynamic Weights` touches governance scoring and priority formation
- `Knowledge Compilation` touches the boundary between source, summary, and authority

These are not small beta-era edits.

### Reason 3: Premature architecture cleanup can erase useful pressure

Right now the repo is still teaching us:

- where operators get confused
- what evidence is actually missing
- which seams hurt in real use versus only in design review

If we extract the center too early, we risk freezing a theoretical answer before the beta lane has taught us enough.

## 4. Program Questions

This dormant line should answer four questions before it ever becomes active implementation work.

### Q1. What is irreducible Domain Core?

Candidate core content:

- axioms
- tension math
- vow contract
- honesty predicate
- perhaps a minimal governance-decision contract

Non-core candidates that may belong in adapters or services:

- session protocol packaging
- dashboard shaping
- packet assembly
- storage backends
- trace transport

### Q2. What should "dormant" mean?

`Dormant` should not mean:

- deleted
- forgotten
- inert forever

It should mean:

- preserved in indexed form
- absent from default attention
- eligible for wake-up by explicit trigger

The first challenge is to define those triggers honestly.

### Q3. Can weights become governance output without becoming mysticism?

Dynamic weights are attractive because they move value-priority from hidden constants into visible governance output.

They are dangerous because they can also become:

- opaque pseudo-wisdom
- untestable priority drift
- a new place to smuggle assumptions

So the question is not "can we make weights dynamic?"

It is:

`what evidence and audit trail would make dynamic weighting more honest than fixed weighting?`

### Q4. Can a knowledge-compilation layer reduce surface count rather than inflate it?

This line is only worth activating if it can replace scattered residue.

If it merely adds:

- `raw`
- `compiled`
- `tensions`
- `artifacts`

on top of everything else already present, it fails its own stated philosophy.

Activation therefore requires a replacement map, not just a new folder scheme.

## 5. Workstreams For The Future Active Version

These are not active now.
They are the bounded workstreams that would exist after wake-up.

## Workstream A: Domain Core Boundary Map

Goal:

- identify what must stay central and what should become ports/adapters

Allowed in dormant mode:

- dependency mapping
- file-size / responsibility mapping
- conceptual boundary diagrams

Not allowed in dormant mode:

- runtime module extraction
- moving live logic out of `unified_pipeline.py`
- claiming the new center is already proven

### Early output shape

- one boundary map
- one "keep / move / undecided" inventory
- one migration-risk table

## Workstream B: Dormancy Semantics

Goal:

- define `active / dormant / archived` as honest state semantics

Recommended first activation target:

- plans and capabilities

Not recommended as first activation target:

- memory scoring
- governance runtime
- automatic self-preservation logic

Why:

- plans and capabilities are easier to inspect
- runtime dormancy semantics are much easier to mythologize

### Early output shape

- state-transition table
- wake-trigger grammar
- examples of dormant-but-indexed versus archived

## Workstream C: Dynamic Weights As Governance Output

Goal:

- decide whether any scoring weights should become generated governance output

Required before activation:

- visible inputs
- audit trail
- replayable examples
- human ack/apply boundary

Non-goal:

- replacing human judgment with opaque "weight wisdom"

### Early output shape

- one bounded spec
- example traces showing why a weight shift happened
- no runtime mutation until trace quality is good enough

## Workstream D: Knowledge Compilation Replacement Plan

Goal:

- create a path where compiled knowledge reduces, rather than increases, system sprawl

Required before activation:

- inventory of existing residue:
  - handoff
  - traces
  - observer summaries
  - self_journal
  - footprints
- explicit replacement or retirement map

Failure condition:

- if net surface count goes up and no old surface becomes unnecessary, this workstream should stay dormant

### Early output shape

- source taxonomy
- replacement map
- advisory / canonical boundary table
