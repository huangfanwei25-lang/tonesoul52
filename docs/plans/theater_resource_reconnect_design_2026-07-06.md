# 資源接回設計 — 讓三資源重新有意義(2026-07-06,目標模式)

> 性質:**設計,進迴圈受 grill + codex 審**。owner 決定:接回資源(不走整套城基、不收掉資源)。
> 背景:結局 MVP 後三資源(醫療/能源/信任)只餵 HUD/預覽/下載,**不影響結果** → 玩家會問
> 「這些數字幹嘛的」。本設計把它接回「有意義」,但**不重蹈覆轍**(不變成可最佳化的分數、
> 不重造 energy>=5 那種必 dark 的門檻)。
> 方法:目標模式迴圈——本設計 → grill 多角度 → codex 跨模型 → 實作 → codex 審碼 → 測試 → 上線。

## 〇、原則(不可違)

資源**不是輸贏分數**,是城市的**物質現實**(接世界書「每次分流救一些人、犧牲一些人」)。
- 道德「形狀」= 錨鏈守信(結局家族,MVP 已定,**本設計不碰**)。
- 物質「質地」= 資源(中途約束 + 尾聲點名)。
兩層都算數,沒有一個能被 min-max 成「正解」。**restraint 守門**:任何讓玩家算出刷資源最優路的設計,拒絕。

## 一、三個接點

### 接點 A — 結局尾聲讀資源(便宜、安全;先做)
「你保住了什麼 / 犧牲了什麼」讀**最終資源狀態**,點名具體物質後果:
- 醫療 ≤3:「白線醫療站在你任內數次拒收」/ 醫療 ≥7:「傷者名單那晚,大多接住了」
- 能源 ≤3:「灰區東三管線多次斷電」/ 能源 ≥7:「城的燈,大致沒滅」
- 信任 ≤3:「灰區不再聽你說話」/ 信任 ≥7:「他們還願意等你開口」
資源進**責任紀錄**(遊戲全部重點),但**不判善惡、不決定結局家族**——只是把物質代價點名。
**實作**:結局頁三問的「犧牲」段 + 尾聲加一段資源 readout(讀既有 S.resources,零新狀態)。

### 接點 B — 能源閘第三路(強;順手解「能源只扣」)
需要電力的**第三條路(火花)**在能源低時不可用或雙倍代價:
- 能源 ≥ 門檻(建議 4):第三路正常。
- 能源 < 門檻:第三路標「電力不足」——可選但付額外具名代價(或關閉,grill 決定哪個)。
**為什麼這個接點最關鍵**:它讓「能源掉」有**後果**——掉越多、越沒有第三條路。你原本的抱怨
(能源只扣、很煩)在這裡從「無意義的惱人」變成「真實的取捨:你為了眼前的人燒了能源,
後面就少一條創造性的出路」。**一刀解兩題**。

### 接點 C — 低資源在選項上標警告(輕版;先只做顯示,不做分支)
低資源時,相關選項的 risk_disclosure 多一行動態警告(「醫療已低:此選項會讓某站失去救治能力」)。
**先只做「看得見的約束」,不做「選項因資源消失」的全分支**(那是大內容工程,等真人試玩再擴)。

## 二、範圍(這一刀做什麼、不做什麼)

**做**:A(尾聲讀資源)+ B(能源閘第三路)+ C 輕版(低資源警告顯示)。
**不做**(明列防 scope creep):
- 不改結局家族判定(錨鏈守信,MVP 已定)。
- 不做「選項因資源全分支消失」的大內容工程(C 只做顯示)。
- 不動存檔 schema(A/B/C 全讀既有 S.resources,無新欄位 → 舊存檔相容)。
- 不碰城基大刀(那是另一個 owner 決定,已擱置)。

## 三、預期會被 grill/codex 電的點(先自標)

1. **A 的門檻(≤3/≥7)會不會又變成可最佳化?** — A 只點名不判勝負,但玩家可能為了漂亮 readout 刷資源。需想:是否只點名、不給任何機械好處。
2. **B 關閉 vs 加價?** — 關閉第三路較狠但可能讓卡關;加價較軟。grill 決定。
3. **B 的門檻與能源起始(5)交互** — 起始 5、門檻 4,預設有第三路;能源掉到 3 才失去。這個曲線對不對要真人試玩。
4. **C 的警告文字從哪來?** — events.json 沒有資源條件警告欄位;要嘛硬編通用句、要嘛加欄位(加欄位=內容工),grill 決定。
5. **存檔相容** — A/B/C 讀既有 resources,理論上舊檔相容,但**動手時必驗**(今天 NaN 判例)。
6. **顯示殘留** — 接回後資源在 HUD/預覽/尾聲三處語義要一致,別一處說了算一處不算。

## 四、迴圈計劃(目標模式)

1. 本設計 → **grill 五視角**(讀當前 code 找地雷)。
2. **codex 跨模型審**設計(抓 claude correlated 盲點,owner 指定的多角度)。
3. 依 grill+codex 收斂,**實作** A+B+C輕版。
4. **codex 審碼**。
5. **測試**(結局尾聲 readout、第三路閘門的 regression test)+ 本地 smoke。
6. **上線驗證** → checkpoint 回報 owner。

碰到 owner 決定點(如 B 關閉 vs 加價若涉手感取捨)→ 標記、給建議、繼續能繼續的,不卡整條。
