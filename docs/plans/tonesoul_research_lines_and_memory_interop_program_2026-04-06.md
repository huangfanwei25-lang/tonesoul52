# ToneSoul Research Lines And Memory Interop Program (2026-04-06)

> Purpose: turn the latest external research distillations into a bounded ToneSoul work program that improves cross-agent coherence without reopening mythology-heavy architecture drift.
> Status: accepted planning candidate
> Authority: planning aid. Runtime truth remains in code, tests, and accepted architecture contracts.

---

## Why This Program Exists

Recent research and ecosystem review made three things clearer:

1. ToneSoul already has enough continuity surfaces to share state.
2. ToneSoul still does not make different AI consumers interpret that shared state in the same way.
3. The next gains are no longer mostly about inventing new metaphors.
   They are about:
   - better knowledge layering
   - better cross-agent consumer parity
   - better honesty around prediction vs description

This program exists because the current stack is strong enough that the next bottleneck is:

`shared interpretation, not only shared transport.`

---

## Why These New Lines Exist

### Line 1: Knowledge Layer

Why it exists:

- `R-memory` is hot coordination state
- it is not a proper retrieval index
- retrieval, continuity, and identity still need harder separation

Research pressure:

- `RAG`
- `qmd`
- `llm-knowledge-base`

ToneSoul problem it answers:

- later agents still lack one honest, queryable, health-checked knowledge layer

### Line 2: Predictive-vs-Descriptive Split

Why it exists:

- ToneSoul already exposes descriptive confidence
- but launch posture, validation health, and operational trends will eventually need predictive framing

Research pressure:

- `Self-Consistency`
- `TimesFM`
- existing ToneSoul council realism work

ToneSoul problem it answers:

- too many readers still over-read descriptive agreement as if it were forecast-grade certainty

### Line 3: Bounded Internal-State Observability

Why it exists:

- fail-stop, mirror rupture, anti-fake-completion, and observer-window all imply that some internal strain/drift/readiness states matter

Research pressure:

- Anthropic interpretability work
- emotion-concept functional-state research

ToneSoul problem it answers:

- the system has behaviorally important strain states, but current readouts are still coarser than they could be

### Line 4: Cross-Agent Memory Interop

Why it exists:

- same repo
- same R-memory
- still different Codex/Claude interpretations

That means the missing layer is not just memory transport.
It is:

- consumer parity
- entry parity
- closeout parity
- mutation interpretation parity

---

## Official Reality Check

The current official OpenAI reality is:

- Codex CLI is open source
- Codex is now described across CLI / web / IDE / app
- skills and automation are part of the official direction

What is **not** currently established in official OpenAI sources is:

- a first-party native "Codex inside Claude" interoperability surface
- a shared hidden memory or cognition layer across vendors

ToneSoul should therefore assume:

`cross-vendor coherence must still be built repo-side.`

That means:

- R-memory remains useful
- but R-memory alone is not enough
- consumer contracts must become stronger

---

## Program Thesis

The next mature form of ToneSoul should have:

1. `one bounded hot-memory transport`
2. `one bounded interpretation contract`
3. `one retrieval/knowledge layer that stays separate from identity`
4. `one honest split between descriptive state and predictive operations`

In short:

`stronger bridges, not more magic.`

## Proposed Phase Ladder

### Phase 775: Cross-Agent Memory Consumer Contract

Bounded scope:

- define one repo-native contract for how all agent shells should consume:
  - session-start
  - observer window
  - packet
  - closeout grammar
  - mutation preflight
- focus on interpretation parity, not transport changes

Success criteria:

- Codex, Claude-style, and dashboard consumers can be judged against one shared interpretation contract

Current implementation target:

- one bounded `consumer_contract` visible in session-start, observer-window, and packet surfaces
- one shared first-hop order
- one shared misread-guard set
- no transport rewrite and no fake first-party vendor interop claim

### Phase 776: Claude-Compatible Entry Adapter

Bounded scope:

- define a minimal adapter for Claude-style shells that preserves ToneSoul's first-hop ordering
- keep it repo-native and explicit
- do not pretend official vendor-native interop exists

Success criteria:

- a Claude-like shell can cold-start from a bounded adapter without immediately drifting from Codex's reading order

Current implementation target:

- one repo-native `run_claude_entry_adapter.py`
- built from Tier-1 session-start, not from raw packet
- preserves the same first-hop order already established by the consumer contract
- explicitly avoids any claim of official first-party vendor interop

### Phase 777: Knowledge Layer Boundary Contract

Bounded scope:

- define:
  - raw sources
  - compiled knowledge
  - exploratory residue
  - operator retrieval surfaces
- explicitly separate retrieval from continuity and identity

Success criteria:

- ToneSoul can explain where future knowledge indexing belongs without colliding with R-memory

Current implementation target:

- one explicit boundary between canonical anchors, hot coordination state, compiled knowledge, exploratory residue, and operator retrieval surfaces
- no fake retrieval runtime yet
- no collapse of retrieval into memory or identity

### Phase 778: Launch Health Trend Readout Design

Bounded scope:

- identify which launch/beta metrics are:
  - descriptive only
  - trendable
  - maybe forecastable later
- do not add predictive numbers yet

Success criteria:

- future launch-health forecasting has a clean home without polluting council confidence today

Current implementation target:

- one bounded `launch_health_trend_posture` shared across packet, session-start import posture, and diagnose
- explicit separation between:
  - `descriptive_only`
  - `trendable`
  - `forecast_later`
- no predictive numbers, no launch probability, no fake calibration
- operator guidance may mention the posture, but it must stay advisory and present-tense

### Phase 779: Bounded Internal-State Observability Reality Check

Bounded scope:

- define which internal states may be responsibly externalized as:
  - strain
  - drift
  - unresolved conflict
  - stop-reason pressure
- keep selfhood claims out

Success criteria:

- ToneSoul gets a cleaner observability lane without pretending mapped states equal subjective experience
